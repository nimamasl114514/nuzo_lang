## 二、发现的 Bug（按严重度排序）

### 🔴 P0 — 致命缺陷

#### B1. Windows 默认栈溢出（新发现！）

**现象**：在 Windows 上用默认 1MB 线程栈运行任何 Nuzo 程序（包括最简单的 `println("hello")`）都会 `STATUS_STACK_OVERFLOW`。

**根因**：`Compiler::compile()` 内部的递归下降解析器 + IR 构建器在 debug 模式下需要 ~4MB 栈空间。release 模式下稍好但仍超过 1MB。

**影响**：**所有 Windows 用户默认无法运行 Nuzo 程序**。

**修复方向**：
- 增大默认线程栈（link args `/STACK:8388608`）
- 或将深层递归改为显式栈迭代

**验证数据**：
- 1MB 栈 → 崩溃
- 2MB 栈 → 崩溃  
- 4MB 栈 → 有时崩溃
- 8MB 栈 → 正常

---

### 🟠 P1 — 高优先级 Bug

#### B2. 异常处理 IR 构建未实现

**现象**：所有 `try/catch/out/keep` 语法均编译失败。

**错误信息**：
```
IR build failed: Exception handling not yet implemented in IR builder
```

**影响**：异常处理系统完全不可用（3/3 测试失败）。

#### B3. `typeof` 返回值不一致

**现象**：`typeof(42)` 返回 `"integer"` 而非文档描述的 `"number"`。

**实际输出**：`["integer", "string", "bool", "nil", "array", "dict"]`

**影响**：破坏 API 契约，依赖 `typeof` 的类型检测代码会出错。

#### B4. 字典键类型限制

**现象**：字典键只接受字符串，整数键报 `type mismatch: expected string key, got integer`。

**影响**：无法用 `d[0]`, `d[1]` 作为键，只能用 `d["0"]`, `d["k1"]`。

#### B5. 中文关键字 `打印` 未注册

**现象**：`打印("hello")` 报 `undefined variable '打印'`。但 `如果`/`否则`/`真`/`假` 等关键字正常。

**影响**：中英文双语支持不完整。

---

### 🟡 P2 — 中优先级 Bug

#### B6. 幂运算符结合性错误

**现象**：`2 ** 3 ** 2` 报解析错误 `unexpected token: *`。`**` 不支持链式使用。

**影响**：`**` 运算符只能单次使用，不能 `a ** b ** c`。

#### B7. `break` 带值行为异常

**现象**：`loop { break value }` 中 `break` 的值丢失，返回 `nil`。

**错误**：`type mismatch: expected number, got nil`

#### B8. 链式比较语义错误

**现象**：`1 < 2 < 3` 解析为 `(1 < 2) < 3` = `true < 3`，导致类型错误。

**预期**：应该像 Python 一样语义化为 `1 < 2 && 2 < 3`。

#### B9. 数字+字符串类型检查缺失

**现象**：`1 + "hello"` 不报类型错误，静默返回空输出。应报编译时或运行时类型错误。

---

### 🟢 P3 — 低优先级

#### B10. 高阶函数语法限制

**现象**：`fn apply(fn, val) { return fn(val) }` 报解析错误。函数参数名 `fn` 与关键字冲突。

#### B11. 取模语法不一致

**现象**：`-10 % 3` 报解析错误。负数取模的括号处理有问题。

#### B12. 嵌套块属性语法

**现象**：`{ a = 1; { c = a + 2; c * 3 } }` 中的嵌套块被误解析为字典属性。

## 三、性能瓶颈分析

### 3.1 基准测试结果

| 测试 | 编译(ms) | 执行(ms) | 总计(ms) |
|------|----------|----------|----------|
| 递归fib(30) | 0.27 | 8988.8 | **8989.1** |
| 尾递归count(10万) | 0.45 | 6.5 | 7.0 |
| while累加(100万) | 0.47 | 11.6 | 12.1 |
| 数组push(1万) | 0.35 | 0.9 | 1.2 |
| 字典插入(1千) | 0.64 | 2.2 | 2.8 |
| 字符串拼接(1千次) | 0.54 | 2.5 | 3.0 |
| 嵌套循环(100x100) | 0.42 | 1.4 | 1.8 |
| 闭包调用(1千次) | 0.40 | 1.1 | 1.5 |
| 长算术链 | 0.03 | 0.01 | 0.05 |
| 非尾递归深度(20) | 0.32 | 0.06 | 0.38 |

### 3.2 关键瓶颈

#### 🔴 瓶颈 1：非尾递归调用极慢

`fib(30)` 执行 8989ms，而 Rust 原生只需 1.6ms。**慢 5600 倍**。

**根因分析**：
- `fib(30)` 产生 2,692,537 次函数调用
- 每次调用涉及：CallFrame 创建、寄存器保存/恢复、GC 屏障检查
- 非尾递归没有 TCO，每个调用帧都保留在栈上

**对比**：尾递归 `count(10万)` 仅需 6.5ms，TCO 优化效果显著。

#### 🟡 瓶颈 2：编译开销偏高

简单程序编译需 0.3-0.6ms，对于解释型/字节码 VM 来说偏高。

**根因**：
- `eprintln!` IR dump 在 debug 模式输出到 stderr（即使 release 也编译了该路径）
- 多层 IR 优化 Pass（常量折叠、死代码消除）

#### 🟡 瓶颈 3：循环基础开销

`while` 累加 100 万次需 12ms，Rust 原生 < 0.001ms。

**根因**：
- 每次迭代都是解释执行（无 JIT）
- 寄存器式的 VM 指令派发有固定开销

### 3.3 性能对比总结

| 场景 | Nuzo | Rust 原生 | 倍数 |
|------|------|-----------|------|
| fib(30) | 8989ms | 1.57ms | **~5700x** |
| while 100万 | 12ms | <0.001ms | **>12000x** |
| 数组push 1万 | 0.9ms | 0.06ms | ~15x |
| 字符串拼接 1千 | 2.5ms | 0.005ms | ~500x |
| 嵌套循环 100x100 | 1.4ms | <0.001ms | **>1400x** |

**总体**：Nuzo 的解释执行比 Rust 原生慢 **1000-10000 倍**，这在字节码 VM 中是正常范围（Python 也有类似差距）。

| 尾递归优化 | ✅ TCO 工作正常，支持 20000 层深度 |

## 五、建议修复优先级

1. **立即修复**：Windows 栈溢出（影响所有用户）
2. **高优先**：异常处理 IR 实现、`typeof` 返回值修正
3. **中优先**：字典整数键支持、`打印` 注册、链式比较语义
4. **低优先**：`**` 结合性、`break` 带值、编译性能优化
